# i18n / multi-language

The library ships neutral English defaults for every user-facing string in
the cart drawer and the checkout, plus ready-made translation presets
for languages that are common across 1click stores. Stores pick their
active language by passing the matching labels object to the provider —
at provider construction time or dynamically based on the active locale.

## Single-language stores

Most stores operate in one language. The store's layout imports the
matching preset once and mounts the provider with it:

```tsx
// apps/my-store/src/app/layout.tsx
import {
  CartDrawerProvider,
  bulgarianCartDrawerLabels,
} from "@1click/ui/cart-drawer"
import {
  CheckoutProvider,
  bulgarianCheckoutLabels,
} from "@1click/ui/checkout"

export default function RootLayout({ children }) {
  return (
    <CheckoutProvider labels={bulgarianCheckoutLabels}>
      <CartDrawerProvider cart={cart} labels={bulgarianCartDrawerLabels}>
        {children}
      </CartDrawerProvider>
    </CheckoutProvider>
  )
}
```

That's it. Every cart drawer and checkout primitive renders in Bulgarian.

## Partial overrides

Stores that want Bulgarian as a baseline but need to tweak specific
strings (e.g. a store named "FamShop" replacing "Your cart" with
"FamShop cart") can spread the preset and override:

```tsx
<CartDrawerProvider
  cart={cart}
  labels={{
    ...bulgarianCartDrawerLabels,
    yourCart: "Вашата FamShop кошница",
  }}
>
```

The context merges the override onto the full set, so only that one
string changes. Every other string stays Bulgarian.

## Multi-language stores

When a single store needs to serve multiple languages (language switcher
in the header, or locale-based routing), map the active locale to the
matching labels object at render time.

The library already exposes locale state helpers from
`@1click/ui/data/locale-actions`:

```tsx
import { getLocale } from "@1click/ui/data/locale-actions"
import {
  CartDrawerProvider,
  bulgarianCartDrawerLabels,
  defaultCartDrawerLabels,
} from "@1click/ui/cart-drawer"
import {
  CheckoutProvider,
  bulgarianCheckoutLabels,
  defaultCheckoutLabels,
} from "@1click/ui/checkout"

const cartDrawerLabelsByLocale = {
  bg: bulgarianCartDrawerLabels,
  en: defaultCartDrawerLabels,
}

const checkoutLabelsByLocale = {
  bg: bulgarianCheckoutLabels,
  en: defaultCheckoutLabels,
}

export default async function RootLayout({ children }) {
  const locale = (await getLocale()) || "bg"

  const cartLabels =
    cartDrawerLabelsByLocale[locale] ?? bulgarianCartDrawerLabels
  const checkoutLabels =
    checkoutLabelsByLocale[locale] ?? bulgarianCheckoutLabels

  return (
    <CheckoutProvider labels={checkoutLabels}>
      <CartDrawerProvider cart={cart} labels={cartLabels}>
        {children}
      </CartDrawerProvider>
    </CheckoutProvider>
  )
}
```

When the user changes language, the store calls `updateLocale("en")`
(also from `@1click/ui/data/locale-actions`), which:
1. Updates the locale cookie.
2. Syncs the active cart's `locale` field so server-side emails,
   admin displays, and the backend render in the new language.
3. Revalidates cached product / category / collection data.
4. Re-renders the layout, which picks up the new locale and passes the
   new labels object to the providers.

## Adding a new language

When a client store needs a new language that isn't shipped yet:

1. Add `src/cart-drawer/labels-<code>.ts` and `src/checkout/labels-<code>.ts`
   exporting a typed `CartDrawerLabels` / `CheckoutLabels` object.
2. Re-export from the cart-drawer and checkout barrels.
3. Add subpath exports in `package.json`.
4. Tag a new release. Stores that want the new language bump their
   library version and map the locale in their `labelsByLocale` object.

The type system enforces that every field in `CartDrawerLabels` /
`CheckoutLabels` is present — missing translations fail typecheck.

## What belongs in labels vs content

Labels are **UI chrome** — button text, form field labels, placeholders,
status messages, tooltips. Things the library's components render and
that always look identical in structure.

They are NOT:
- Product titles, descriptions, or variant names — these come from the
  Medusa backend with `locale` set on the cart, and are already localized
  server-side.
- Store branding (logo, store name, tagline) — store-specific, lives in
  the store's own `src/`.
- Marketing copy, hero sections, blog content — all unlocked store code.
- Currency / number formatting — handled by `convertToLocale` from
  `@1click/ui/lib/money`, which uses Intl.NumberFormat with a locale
  argument (default `"en-US"`).

When in doubt: if it's a fixed string rendered by a library primitive,
it's a label. If it's user-generated or store-generated content, it's
not.
